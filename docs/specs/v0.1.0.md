# Agent Base v0.1.0

Status: Accepted design
Date: 2026-06-27

## Outcome

Agent Base v0.1.0 is a single-owner, localhost-hosted research platform. The Owner can configure a reusable Agent, ground a Task with uploaded documents and public web evidence, approve a proposed Research Plan, inspect the Run, and accept or revise an evidence-linked Report.

The release is local-hosted but not offline. MiniMax and Tavily remain external services and require internet access.

## Product boundaries

V0.1.0 includes:

- one implicit Owner and Workspace;
- guided Agent authoring and immutable Agent Versions;
- Projects with reusable uploaded Sources;
- Tasks with multiple immutable Run attempts;
- a built-in research runtime;
- MiniMax Token Plan model access using an Owner-supplied key;
- Tavily web discovery using an Owner-supplied key;
- guarded static HTML and PDF retrieval;
- a mandatory Research Plan approval gate;
- inspectable Run Events and checkpoint recovery;
- evidence-linked Reports with Markdown export; and
- native Windows installation, backup, restore, and upgrades without requiring Docker or separately installed runtime dependencies.

V0.1.0 excludes:

- authentication, multiple owners, invitations, roles, and Project ACLs;
- remote or hosted deployment support;
- executable Workflows;
- MCP and arbitrary Connectors;
- public APIs, service tokens, and a CLI interface;
- model-provider selection;
- scanned-document OCR, images, spreadsheets, presentations, and archives;
- JavaScript browser rendering and authenticated web retrieval;
- PDF or DOCX Report export; and
- product telemetry.
- Docker-based packaging and deployment.
- native end-user distributions for macOS and Linux.

## Canonical user journey

1. Install Agent Base on Windows, start it from the installed application, and open the loopback-bound web application.
2. Enter MiniMax Token Plan and Tavily Credentials in Settings. Agent Base validates and encrypts them.
3. Use the seeded general research Agent or edit its Agent Draft and publish an Agent Version.
4. Create a Project and upload text PDF, DOCX, Markdown, or TXT Sources.
5. Wait for selected Sources to finish parsing and indexing.
6. Create a Task with a research goal, Report language, selected Project Sources, and an explicit web-research choice.
7. Start a Run using a published Agent Version.
8. Review the generated Research Plan. Approval continues the Run; cancellation ends it.
9. Inspect live Run Events while the Agent searches, retrieves, records evidence, and drafts.
10. Review the completed Report and its Source Excerpts.
11. Accept the Report to complete the Task, or supply revision feedback to start another Run.
12. Export the accepted Report as Markdown with its source appendix.

## Agent configuration

An Agent is a stable identity with one mutable Agent Draft. Publishing creates the next sequential immutable Agent Version.

The guided Agent Draft contains:

- purpose;
- operating instructions;
- research method;
- Report outline and requirements;
- permission to use Project Sources;
- permission to use web evidence; and
- Run limits that may only reduce the platform maximums.

Setup creates and publishes a general research Agent so the first Task does not require authoring work.

## Sources and retrieval

### Uploaded Sources

- Accept text-based PDF, DOCX, Markdown, and TXT.
- Limit a file to 25 MB and a Project to 100 active Project Sources.
- Reject image-only PDFs and unsupported or mismatched file types.
- Parse asynchronously and expose `pending`, `processing`, `ready`, and `failed` states.
- Chunk normalized text into approximately 400-token passages with 60-token overlap while retaining page, heading, and paragraph locators where available.
- Create 384-dimensional embeddings locally with the quantized `Xenova/multilingual-e5-small` ONNX model.
- Combine PostgreSQL simple full-text and trigram matching with vector similarity through reciprocal-rank fusion, returning at most 12 chunks per retrieval.
- Do not send uploaded content to an embedding service or Tavily. Selected passages are sent to MiniMax when required for the Run.

Deleting a Project Source removes its original file and reusable chunks. Historical Source Excerpts already retained by Runs remain available until the entire Project is permanently deleted.

### Web Sources

- Tavily search uses `basic` depth, at most eight results, no generated answer, and no raw page content.
- Search results are discovery candidates and cannot be cited directly.
- The fetcher accepts public HTTP or HTTPS static HTML and text PDF only.
- Revalidate DNS and policy after every redirect; block loopback, private, link-local, multicast, and metadata-service addresses.
- Allow at most three redirects, a ten-second response timeout, 10 MB HTML, and 25 MB PDF.
- Respect robots directives and do not bypass login, paywall, or access controls.
- Extract readable content without executing JavaScript.
- Persist only metadata and exact Source Excerpts used by the Report; discard transient full-page content after the Run.

All Source content is untrusted evidence. It is delimited separately from instructions and cannot add tools, change policies, or authorize actions.

## Task, Run, and Report model

A Task contains its goal, output language, selected Project Sources, web permission, and lifecycle state: `open`, `in_review`, `completed`, or `archived`.

Starting a Run snapshots:

- the Task goal and language;
- selected Source identities and content revisions;
- web permission;
- revision feedback, if any;
- the Agent Version;
- model identifier; and
- effective limits.

Run states are:

- `queued`;
- `planning`;
- `awaiting_approval`;
- `running`;
- `paused_credentials`;
- `paused_quota`;
- `succeeded`;
- `failed`; and
- `cancelled`.

Only one Run executes at a time. Further Runs remain queued. Source ingestion has a separate concurrency-one lane.

Each Run is capped at:

- 20 MiniMax model turns;
- 10 Tavily searches;
- 30 page fetches; and
- 15 minutes of active execution time.

Waiting for approval or provider recovery does not consume active execution time.

The Research Plan is generated before evidence tools are available. Approval records who approved and when, then unlocks the Run. Changing the plan requires cancelling and starting a new Run.

The runtime exposes only these evidence operations:

- search selected Project Sources;
- search the public web;
- retrieve an eligible public page;
- retain an exact Source Excerpt; and
- submit a structured Report.

A Report contains ordered sections and typed blocks: `factual`, `synthesis`, `recommendation`, or `limitation`. Factual and synthesis blocks require at least one valid Source Excerpt; recommendation and limitation blocks may omit citations unless they include factual premises. Submission fails when a citation is missing, unknown, or points outside the Run's evidence set.

Run completion creates one immutable Report and moves the Task to `in_review`. Revision feedback creates a new Run using the previous Report as an explicit input. Accepting a Report records a Task Acceptance and moves the Task to `completed`.

## Runtime history and recovery

The Run Event stream records:

- state transitions;
- Research Plans and approvals;
- model-visible messages;
- tool calls and sanitized results;
- Source Excerpts and citations;
- provider usage and enforced limits;
- retries, pauses, errors, and cancellation; and
- Report creation and acceptance.

Provider-specific hidden reasoning is neither persisted nor displayed.

The worker checkpoints after each committed model or tool step. A restarted worker leases unfinished work and resumes from the latest checkpoint. External API calls are at-least-once: a crash after a service accepts a request but before Agent Base commits its response may repeat that request. This is accepted in v0.1.0 because all tools are read-only, though a repeated request can consume quota.

Invalid or expired Credentials pause a Run until the Owner replaces the Credential. Provider quota exhaustion pauses the Run until the Owner resumes it after capacity returns. Cancellation is terminal and never creates a Report.

## Technical architecture

### Foundation

- Node.js 24 LTS;
- Next.js 16 and React 19;
- strict TypeScript;
- pnpm workspaces;
- Drizzle for application-owned database access and migrations;
- PostgreSQL 18 with pgvector 0.8.2;
- `pg-boss` for durable jobs;
- Transformers.js with pinned quantized ONNX weights;
- Biome for formatting and linting;
- Vitest for unit and integration tests; and
- Playwright for browser acceptance tests.

### Repository shape

- `apps/web`: local UI, internal route handlers, and Server-Sent Event delivery;
- `apps/worker`: Source ingestion and Run execution;
- `packages/domain`: domain types, state transitions, and invariants;
- `packages/application`: use cases and outbound ports; and
- `packages/infrastructure`: PostgreSQL, filesystem, provider, parser, retrieval, and queue adapters.

Core logic must not import Next.js, provider SDKs, or database clients.

### Ports

Define stable internal ports for:

- `ActorContext`;
- `ModelProvider`;
- `SearchProvider`;
- `PageFetcher`;
- `EmbeddingProvider`;
- `BlobStore`;
- `RunQueue`; and
- aggregate repositories.

V0.1.0 supplies singleton-local, MiniMax, Tavily, guarded HTTP, local ONNX, local filesystem, and PostgreSQL implementations. These are internal interfaces, not a public extension API.

### Persistence

Persist Owner, Workspace, Credential, Agent, Agent Draft, Agent Version, Project, Source, Source Chunk, Task, Run, Research Plan, Run Event, Source Excerpt, Report, and Task Acceptance records.

Every root aggregate carries Workspace ownership and every user-originated action carries Owner identity, even though both are singleton values in v0.1.0.

### Internal HTTP boundary

Use validated JSON route handlers for Settings, Agents, Projects, Sources, Tasks, Runs, plan approval, cancellation, resume, revision, Report acceptance, and export. Use a cursor-based Server-Sent Event endpoint for Run updates. These endpoints have no public compatibility promise in v0.1.0.

## Native installation and security

The Windows distribution installs and manages the web process, worker, and a private PostgreSQL/pgvector instance. It must not require the Owner to preinstall Docker, Node.js, pnpm, PostgreSQL, or pgvector. PostgreSQL is reachable only by Agent Base processes, and the web port binds only to `127.0.0.1`.

Local initialization creates an Agent Base application-data directory under the Owner's Windows profile containing:

- generated database configuration;
- the encryption master key;
- PostgreSQL data;
- uploaded Source bytes;
- Report exports; and
- local operational logs.

Credentials are encrypted with AES-256-GCM and never returned after entry. Logs, Run Events, errors, and exports must redact secrets. Mutating requests require same-origin Host and Origin validation plus CSRF protection to defend the local service from hostile web pages and DNS rebinding.

Application processes run with the Owner's standard Windows privileges and do not require administrator privileges after installation. Upload handling performs MIME sniffing, archive-expansion limits for DOCX, parser timeouts, output-size limits, and HTML sanitization. No product telemetry is emitted.

Provide installed controls for initialization, start/stop, health inspection, database migration, backup, restore, and uninstall. Upgrades preserve application data and run required migrations. A backup includes PostgreSQL, file storage, and the encryption master key and must be treated as a secret-bearing artifact.

Windows is the primary and supported v0.1.0 Owner platform. Development is supported on macOS through repository-local development commands and dependencies. Native macOS and Linux end-user distributions, Docker packaging, and remote exposure are deferred beyond v0.1.0.

## Migration path to scale

Future scale work is additive around the v0.1.0 domain:

- replace singleton `ActorContext` with authentication and authorization;
- expose existing Workspace ownership through membership and Project ACLs;
- replace the filesystem Blob Store with S3-compatible storage;
- configure per-owner and instance concurrency instead of the singleton Run lease;
- operate multiple stateless web and worker replicas;
- move secrets to a managed secret store;
- add a versioned public API and service identities; and
- add provider and Connector registries only when their contracts are understood.
- optionally add Docker packaging without making Docker a prerequisite for the Windows distribution.

Remote binding must not be enabled before authentication, authorization, secure secret management, and deployment hardening exist.

## Verification

### Automated tests

- Unit-test domain state machines, immutability, limits, approval rules, citation validation, encryption, URL policy, chunking, fusion ranking, and Report rendering.
- Integration-test migrations, pgvector retrieval, `pg-boss` leasing and retries, filesystem lifecycle, ingestion, deletion semantics, and checkpoint recovery.
- Contract-test MiniMax and Tavily adapters with fixtures covering streaming, tool calls, malformed responses, invalid Credentials, quotas, rate limits, and timeouts.
- Keep live MiniMax and Tavily smoke tests opt-in so automated tests do not consume Owner quota.
- Use Playwright for setup, Agent publication, all upload formats, Task creation, plan approval, restart recovery, Report review, revision, acceptance, and Markdown export.
- Security-test prompt injection, malicious URLs, redirect rebinding, oversized inputs, malformed PDF and DOCX files, Markdown sanitization, CSRF, and secret redaction.

### Release acceptance

Release v0.1.0 only when:

- a clean Windows installation reaches a healthy loopback-bound UI without Docker, manual database work, or separately installed runtime dependencies;
- installation, upgrade, backup, restore, and uninstall are exercised on a supported Windows environment;
- the Owner can validate and store MiniMax and Tavily Credentials;
- every supported Source format indexes and becomes searchable;
- a combined uploaded-plus-web research Task reaches plan approval, Report review, revision, acceptance, and export;
- every factual and synthesis Report block resolves to retained Source Excerpts;
- queued and active Runs survive worker restart from committed checkpoints;
- invalid Credentials, provider quotas, cancellation, fetch rejection, and ingestion failure produce recoverable, explicit states;
- backup and restore preserve Sources, Run history, Reports, and usable encrypted Credentials; and
- no Credential appears in plaintext persistence, logs, Run Events, browser responses, or exports.
